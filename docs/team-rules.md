# 🚀 팀 협업 가이드 (Git & PR)

> [!IMPORTANT]
> 원활한 협업을 위해 아래 규칙을 반드시 준수해 주세요.

## 📌 PR(Pull Request) 전 필수 확인 사항

1. **최신 상태 유지**: 본인 브랜치에 `develop`(기본 브랜치)을 먼저 **Pull** 받은 후 작업을 진행하세요.
2. **빌드 에러 확인**: Push 전 로컬에서 빌드 에러가 없는지 반드시 확인하세요.
   - JUnit, Swagger, Postman 등의 테스트 결과 **스크린샷**을 PR에 첨부해 주세요.
   - GitHub의 CI/CD 진행 상황을 반드시 확인하세요.
3. **충돌(Conflict) 해결**: Merge 시 컨플릭트가 발생할 경우 작업자가 직접 해결하며, 리뷰가 완료되면 본인이 직접 **Merge** 합니다.

---

## 🌿 브랜치 생성 및 관리 규칙

- **브랜치 보호 룰**: `develop` 직접 Push 금지 + 최소 **2명 이상의 리뷰**가 있어야 Merge 가능합니다.

1. **이슈 생성 후 브랜치 생성**: GitHub Issue 생성 후 관련 브랜치를 만드세요.
2. **동기화**: `develop` 브랜치로 이동 후 최신 코드를 **Pull** 받고 작업을 시작하세요.
3. **브랜치 생성 위치**: 반드시 최신화된 `develop` 브랜치에서 새 브랜치를 만드세요.

---

## ⚠️ 절대 금지 및 주의 사항

- **강제 푸시 금지 (`--force`)**: 어떤 경우에도 강제 푸시는 허용되지 않습니다.
- **직접 푸시 금지**: `develop` 브랜치로 바로 Push 하지 마세요.
- **Git 꼬임 발생 시**: 반드시 **담당자와 상의**하세요. (새로 클론하거나 재연결하는 방법을 권장합니다.)

### 🧨 설정 파일 보안 (application.yaml)

**설정 파일은 절대 레포지토리에 올리지 마세요.**

1. `.gitignore`에 등록되어 있는지 확인하세요.
2. 이미 추적되고 있다면 아래 명령어로 캐시를 지우세요.
   ```bash
   git rm -r --cache .
   ```

## 📝 커밋 컨벤션 (Commit Convention)

| 태그           | 설명                                 |
| :------------- | :----------------------------------- |
| **`Feat`**     | 새로운 기능 구현                     |
| **`Mod`**      | 코드 및 내부 파일 수정               |
| **`Add`**      | 부수적인 코드, 파일, 라이브러리 추가 |
| **`Fix`**      | 버그 및 오류 해결                    |
| **`Refactor`** | 코드 리팩토링                        |
| **`Docs`**     | 문서 작업 (README 등)                |
| **`Chore`**    | 단순 패키지/변수명 변경 등 작은 작업 |
